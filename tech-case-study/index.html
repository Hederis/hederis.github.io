Case Study

Mid-Sized Publisher Implementing an Automated Workflow

In this case study, we consider the example of an educational publishing company, interested in streamlining their entire book production workflow. They’ve heard about “Digital First” workflows and are interested, but they don’t have a clear idea of how to bring about this transformation to their organization. This company uses Microsoft Word Styles to tag their manuscripts and have been laying out their print books in InDesign.They make adjustments to the InDesign file before exporting to EPUB, but they still have to do some extra cleanup on the EPUB files to make them meet internal and external requirements. They’d like to spend less time adjusting and validating their EPUB files, and they’d also like to be able to generate HTML files for the Web at the same time that they create EPUB and print-ready files.

Potential benefits of choosing an automated workflow

The publisher knows that going “Digital First” can’t just be about making clean EPUB and HTML files--their investment in any new processes has to make sense for their bottom line, reducing composition time and costs and hopefully reducing time to market for their content across formats. Because they are in the educational space, meeting accessibility standards is crucial, and they expect any new process to consistently incorporate accessibility standards into every file. Some of the things they’re hoping to gain are:

  • Reduced composition costs
  • Reduced composition time, leading to potential increase in speed to market
  • Simultaneous print and ebook conversion
  • Simultaneous conversion to HTML, which provides a path for using the content in a variety of different ways
  • Files that meet all validation and accessibility requirements

Challenges to consider when choosing an automated workflow

The production director is in favor of the Digital First initiative, but she is concerned that it could result in an automated “black box” workflow that will make it difficult for the production team to design and fine-tune layouts that match the quality her team is known for. She knows that many automated workflows require custom scripting and technological expertise, and she’s concerned that the publisher is not likely to hire the staff needed to maintain the toolchain and train staff, leaving the burden on an already-stretched production department.

Winning Solution

Use Hederis as an out-of-the-box solution

Their workflow with Hederis:

One-time setup:

  1. The publishing company’s in-house designer designs an interior template within the Hederis system that they can reuse for any number of projects. (Hederis staff available to support/assist with this as needed.)

Steps for each book project:

  1. Publishing company’s editorial staff creates the book project in the Hederis system.
  2. Production staff upload any image files to their Hederis project via the UI.
  3. Editorial staff prepares the book’s Word manuscript according to their established process, and uploads it to the Hederis project via the UI.
  4. The Hederis system maps the manuscript tags to Hederis tags, and then converts the manuscript to print and digital layouts.
  5. Production staff views the layouts in the Hederis user interface, and adjusts the design, line breaks, and page breaks as needed for this specific book.
  6. Publishing company’s production and/or editorial staff makes any required text edits directly in the app.
  7. Production staff exports to PDF, EPUB, and updated Word document (for archival purposes).

Other Options Considered

Use the Hederis API, combined with some custom scripts

An “API” is a way to access and use scripts or technology that someone else owns and maintains. In this option, the publishing company would use the Hederis hosted API for the core technology behind their conversion work without going through the Hederis user interface, and then would additionally develop custom scripts for any missing features that they require. The publishing company pays for maintenance on their custom scripts as needed, but Hederis maintains the API toolchain as part of the built-in costs.

Example workflow using the Hederis API combined with custom scripts:

One-time setup:
  1. Hederis and the publishing company work together to create a design template that can be reused for any number of projects.
Steps for each book project:
  1. Publishing company’s production staff uploads images to the Hederis system, or stores them in the cloud where the API can access them.
  2. Publishing company’s editorial staff prepares the Word manuscript and submits it directly to the API (method TBD—either via command line or a file trigger).
  3. The API (a) maps the manuscript tagging to Hederis tags, (b) converts the book to a fully designed PDF and/or EPUB, and (c) returns a download link to the files.
  4. Publishing company’s editorial and/or production staff reviews the output files using standard programs (Adobe Acrobat, etc).
  5. Publishing company’s production staff details any required design or layout changes in a markup template, and makes any required text edits in the original Word file.
  6. Publishing company’s production staff submits the revised manuscript and design file back through the API for conversion, ad infinitum, until a final layout is approved.

Requirements

  • Publishing company will determine what additional features are needed beyond those provided by the Hederis API
  • Publishing company will determine how users will interact with the API, with the help of a development consultant or firm
  • Publishing company will hire a development firm to develop custom scripts to add features that aren’t currently supported by the API
  • Publishing company will develop training materials for internal staff
  • Publishing company will determine workflow steps

Costs

Estimate between $2k-$10k development costs (depending on how much customization is required, and what kind of API connection tool will be used), plus a per-project $119 fee for using the Hederis API, and then ongoing costs to maintain the custom scripts (either contractually through the development firm, or for new in-house staff).

Pros

  • Publishing company is only responsible for maintaining and supporting specific parts of the toolchain More customizable than the pure out-of-the-box option
  • Much of the tech maintenance and associated costs are the responsibility of Hederis

Cons

  • Won’t be able to start using the tool until the custom scripts and API interactor are built
  • May require some technical learning, to use the API
  • Likely won’t be able to build a user interface (like Hederis or InDesign) without significant financial investment
  • No visual tool for adjusting pagination and end-of-line breaks, making page layout adjustments difficult (but possible)

Create their own scripts from scratch for an in-house automated workflow

With this option, the publishing company would hire a development company to build them a completely custom toolchain, tailored to their books and workflows. They would also likely need to hire someone internally to maintain the toolchain once built, support internal staff, and to implement new features/adapt to workflow changes.

Example workflow with an internal automation toolchain:

One-time setup:
  1. A developer creates a design template in the CSS coding language that can be reused for any number of projects, based on an InDesign file provided by the publishing company.
Steps for each book project:
  1. Publishing company’s production staff uploads images to central server or cloud storage location where the toolchain can access them.
  2. Publishing company’s editorial staff prepares the Word manuscript.
  3. Publishing company’s production staff runs the manuscript through the toolchain (method TBD—either via command line or a file trigger).
  4. The toolchain converts the book to PDF and EPUB, and puts the converted PDF and EPUB files in a specified location on the publishing company’s server.
  5. Publishing company’s editorial and/or production staff reviews the output files using standard programs (Adobe Acrobat, etc).
  6. Publishing company’s production staff details any required design or layout changes in a markup template, and makes any required text edits in the original Word file.
  7. Publishing company’s production staff makes any required text edits in the Word file and runs it back through the toolchain conversion, ad infinitum, until a final layout is approved.

Requirements

  • Publishing company will audit current book files, design standards, and printer/ebook retailer requirements, and determine what features are required by the new toolchain
  • Publishing company will work with development firm to determine how users will interact with the toolchain
  • Development firm will determine framework for the toolchain
  • Development firm will write toolchain scripts
  • Publishing company or external contractor will develop training materials for internal staff
  • Publishing company will determine workflow steps

Costs

Estimate between $10k-$50k upfront for toolchain development by a contractor or development firm (final amount depends on scripting language, feature requirements, and user interaction requirements), and then an investment in at least one full-time developer staff member to maintain the tool and support users. Alternately, publishing company may be able to set up a contract with the development firm to maintain the tool for a fee (though the external developers won’t be able to respond as promptly as an in-house staff member).

Pros

  • Can be fully customized to the publishing company’s existing workflow
  • Publishing company has full control over making updates and bug fixes
  • Fixed lifetime cost (same amount of spend regardless of number of projects)

Cons

  • Won’t be able to start using the tool until it is built
  • Requires internal staff for maintenance, or a maintenance contract with associated fees
  • Publishing company is fully responsible for training and support (unless outsourced for additional cost)
  • Likely won’t be able to build a user interface (like Hederis or InDesign) without significant financial investment
  • No visual tool for adjusting pagination and end-of-line breaks, making page layout adjustments difficult (but possible)
